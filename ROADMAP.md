# Multi-Agent Synthesized Roadmap (2026-05-28)

Generated by sending the family's current state (toolkit v1.3.0, humanizer v2.11.0, translator v1.3.0, authoring-suite v1.3.0) to four independent LLM proxies (kimi/codex/gemini/minimax), collecting 28 raw proposals, then asking each proxy to cross-critique and rank.

## Strong-Consensus Items (all 4 vendors proposed; both structured critics ranked top-6)

### 1. Activate Assets D + E consumer cutover  *[both critics #1/#3]*
**Status:** Toolkit ships `typography-rules.json` (v0.5) and `reader-respect-patterns.json` (v0.5). NO consumer reads them. Half the toolkit's data assets are dormant.
**Target:** humanizer v2.12.0 imports both assets into the lex pass (typography normalization + reader-respect anti-pattern deletion).
**Why:** Closes the largest "shipped but unused" gap. Architecturally validates Assets D + E by giving them a live consumer.

### 2. Cross-validate the 260 v0.12 legacy single-vendor pairs  *[both critics #2]*
**Status:** v0.9.1 + v0.10 cross-vendored the top 50 + 16 disagreements. v0.12 bulk-paired 260 new pairs via minimax only. These 260 are single-vendor + uninspected.
**Target:** toolkit v1.4.0 runs codex confirmation + gemini tiebreaker on the 260 pairs, applying the same v0.9.1/v0.10 decision rule.
**Why:** Closes the data-quality gap. Asset G's 422 pairs would all be multi-vendor validated.

### 3. End-to-end golden regression test  *[both critics #3/#5]*
**Status:** End-to-end demo (`evals/end_to_end_demo.py`) is manual-run only. No CI assertion that the pipeline still produces N Asset G hits, M humanness score, etc.
**Target:** `evals/golden_e2e_test.py` runs the demo against fixed expectations (specific terminology hits, threshold scores, regen counts within bounds).
**Why:** Prevents silent regressions when any sibling ships a new release.

### 4. Domain expansion: business + legal + politics  *[both critics #4]*
**Status:** Asset F/G cover technology + news. Translator/authoring users in business, legal, or politics domains get no terminology hints.
**Target:** toolkit v1.5+ mines Elaph + new corpora for business/legal/politics, then pairs.
**Why:** Translator's customer base extends. Heavy work; mining + pairing per domain ≈ 30 minutes.

### 5. Vendor rotation in `score_text_deep`  *[minimax #1, codex #5]*
**Status:** `score_text_deep` defaults to gemini. Calling it 100 times = 100 gemini calls.
**Target:** humanizer v2.13.0 rotates across {gemini, minimax, codex} per call, with optional `prefer_vendor` override.
**Why:** Reduces single-vendor lock-in for cognitive scoring. Cheap to ship.

### 6. Install bundle / package distribution  *[both critics #6]*
**Status:** Four repos must be cloned separately. No PyPI publish. No one-command setup.
**Target:** Top-level `install_family.sh` (or PyPI metapackage) that clones+verifies all four.
**Why:** Makes the family installable as a single unit for external users.

## Novel Gaps (both critics flagged independently — none in original proposals)

### G1. Unicode normalization contract shared by all 4 siblings
Asset C lex pass uses one tokenization; Stage D uses another; `score_text` counts Arabic chars differently. Shared `arabic_normalize.py` (in toolkit) would canonicalize alif variants, hamza placement, lām-alif, tashkeel handling. Cross-cutting.

### G2. Asset version registry for consumer capability negotiation
Every consumer hardcodes `schema_major == "1"`. A registry would let consumers ask "which versions are compatible?" and toolkit would answer with the matrix. Forward-compat door for v2.0 schema bumps.

### G3. Telemetry: which assets influenced this output
Currently a translation comes back with `stages.A_terminology.matched_count: 11` but no per-output trace of WHICH 11 Asset G entries fired. A per-pair influence log would let users audit "why did the LLM use this AR translation?"

## Execution priority for this loop

1. **Item 1** — Asset D + E cutover → humanizer v2.12.0 (consensus + concrete)
2. **Item 2** — Cross-validate v0.12 pairs → toolkit v1.4.0 (consensus + reuses existing pipeline)
3. **Item 5** — Vendor rotation → humanizer v2.13.0 (consensus + cheap)
4. **Item 3** — Golden e2e regression → toolkit v1.4.1 or similar (consensus + locks the architecture)

Items 4 (domain expansion), 6 (install bundle), and the novel gaps G1-G3 become the queue for follow-on sessions.

Of the 28 raw proposals from the 4 vendors, structured cross-critique (codex + minimax) converged on the same 6 items in different orders. Kimi and gemini critics returned prose rather than structured JSON; I synthesized from the structured pair while reading the prose for novel-gap signal.
